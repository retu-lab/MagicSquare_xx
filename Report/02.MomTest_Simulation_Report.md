**방법:** 역할 분리 Mom Test 시뮬레이션 (3턴)

## 2. 인터뷰 답변 (과거 사실)

- **5월 14일 수요일** 저녁, 실습실에서 과제 PDF 수령 후 손으로 빈칸 두 칸 숫자를 먼저 구함
- Java `MagicSquare` 클래스 작성, `fillBlanks()` → `printGrid()` → `isValid()` 순으로 구현
- `isValid()`는 **행 4개·열 4개** 합만 34인지 확인; **대각선 검증 코드 없음**
- 출력이 깔끔해 “맞겠다”고 판단, **11시 40분쯤 LMS 제출**
- 다음 날 아침 채점 **60점**, 코멘트 **“대각선 검증 누락”**
- `isValid()` 첫 실행부터 제출까지 **약 25분** 소요
- 그 25분 동안 출력본을 노트에 옮겨 **행·열을 손으로 더해 봤고 전부 34**
- **대각선은 그날 밤 손·코드 모두 확인하지 않음**
- IDE에서 `isValid()`가 `true`라 더 건드리지 않음
- **스스로 틀렸다는 걸 알 기회 없음** — 다음 날 9시 과목 시간에 점수 확인 후 처음 인지
- 25분 동안 “이상하다”고 느낀 적 없음; 틀렸다면 `fillBlanks()` 로직·중복 숫자만 의심할 뻔했으나 행·열이 맞아 **“계산은 맞게 한 것 같다”**고만 생각
- **대각선 누락은 채점 코멘트 전까지 한 번도 의심하지 않음**

## 3. 표면 문제 (금지 — 솔루션 섞인 문장)

아래는 인터뷰 중 **직접 나오지 않았지만**, 흔히 튀어나오는 **금지 유형** 문장 예시이다.

- ~~“`isValid()`에 대각선 검사만 추가하면 된다”~~
- ~~“자동 검증 도구가 있었으면 60점은 막을 수 있었다”~~
- ~~“제출 전 체크리스트 UI가 필요하다”~~

> 솔루션·미래·의향이 섞여 있어 Mom Test의 **진짜 문제**로 쓰지 않는다.

---

## 4. 진짜 문제 (한 문장)

**4×4 부분 마방진 과제에서 행·열 합과 `isValid()` 결과가 맞아 보이면, 제출 전까지 대각선을 손·코드 어느 쪽으로도 확인하지 않고 “로직/계산은 맞다”고만 판단한다.**

---

## 5. Mom Test 증거 3줄

1. **5월 14일 밤, `isValid()`가 `true`를 반환하고 행·열 합이 34로 맞아 보여 제출했지만, 코드와 손 계산 모두 대각선은 확인하지 않았다.**
2. **제출 전 약 25분 동안 대각선은 건드리지 않았고, 틀렸을 경우 `fillBlanks()` 로직이나 중복 숫자만 의심할 뻔했다.**
3. **대각선 누락은 LMS 채점 코멘트(“대각선 검증 누락”)를 본 다음 날 아침에야 처음 알게 되었다.**

---

## 6. 멘토 코멘트

이번 **가상 인터뷰**에서 잡힌 것은 “도구가 없어서”가 아니라 **행·열·`isValid()`가 맞아 보일 때 대각선 확인을 미루는 행동**이다.  
STEP 2로 갈 때 “`isValid()`에 대각선 추가” 같은 솔루션 제안보다, **“검증 코드와 손 확인 모두 행·열에 머물고, 대각선은 제출·채점 후에야 드러난다”** 를 전제로 두면 된다.
